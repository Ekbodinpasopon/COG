# GitHub Issue Protocol

## Purpose

Define executable repository work as a GitHub Issue before handing it to Codex.

## Create A GitHub Issue When

- files need to be created
- files need to be edited
- repository structure needs to change
- reviewable work should be tracked

## Issue Structure

Every executable issue should contain:

1. Objective
2. Scope
3. Files to inspect
4. Files to create or update
5. Constraints
6. Safety rules
7. Acceptance criteria

## Writing Rules

- Keep the issue executable.
- Do not mix brainstorming and implementation in the same issue.
- List exact files when known.
- State what must not be touched.
- Include public/private boundaries when relevant.

## Handoff Rule

If repository file changes are required, the GitHub Issue should be convertible into a Codex handoff with minimal rewriting.
